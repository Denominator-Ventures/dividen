## Smart Triage System (Task-First Architecture)
- **Mental Model**: Cards = Projects (containers), Checklist Items = Tasks (atomic work items). Every signal item produces tasks, not cards.
- **Triage Protocol** (system-prompt.ts): 8-step Task-First Routing — EXTRACT TASKS → ROUTE TO PROJECT → ADD TO EXISTING (checklist + artifact) → CREATE NEW PROJECT (sparingly, name as initiative not task) → ASSIGN → QUEUE → LEARN → SUMMARIZE
- **upsert_card** action tag: Levenshtein similarity matching (≥80%) — finds existing card by title before creating
- **link_artifact** action tag: Dual-write system — sets direct FK for built-in types (email, document, recording, calendar_event, contact, comms) AND creates generic CardArtifact record. Custom signal types use CardArtifact only (extensible to ANY artifact type string).
- **CardArtifact model**: Generic join table (cardId, artifactType, artifactId, label, metadata) with unique constraint. Supports any custom signal artifact type (slack_message, github_pr, notion_page, etc.)
- **ChecklistItem source tracking**: sourceType, sourceId, sourceLabel fields — traces WHERE each task came from (which signal, which artifact)
- **Board rendering**: Shows per-card task delegation breakdown `[me:2 divi:3 via-divi:1]`, contributor names with 🟢 for DiviDen users, related count, artifact counts (📧📄🎙️📅👤💬🔗)
- **Signal definitions**: Both `cardTypes` (legacy) and `taskTypes` (new) arrays. UI shows taskTypes. All triage prompts follow EXTRACT → ROUTE → CREATE pattern.
- **Custom signals**: Auto-generate task-first triage prompts with signal-specific artifactType for [[link_artifact:{}]]
- **Queue dedup** (queue-dedup.ts): Levenshtein ≥80% prevents duplicate queue items

## Task Delegation & People Model
- **Assignee types on ChecklistItem**: `assigneeType` field — `self` (operator marks complete), `divi` (AI handles directly), `delegated` (routed to another user's Divi agent)
- **Delegation metadata**: `assigneeName`, `assigneeId` (platformUserId of delegate), `delegationStatus` (pending/accepted/in_progress/completed/declined)
- **Display pattern**: Delegated tasks show as "Name via Divi" so the board feels human, not just "me and AI"
- **People on cards**: `CardContact.involvement` — `contributor` (can take tasks, shown by name) vs `related` (contextual only, shown as count)
- **DiviDen user detection**: `CardContact.canDelegate` auto-set from `contact.platformUserId` presence. 🟢 = DiviDen user who can receive delegated tasks
- **Delegation flow**: Operator → Divi (project manager) → target user's Divi agent (via relay). Divi orchestrates, never bypasses.
- **Kanban query includes contacts**: `contacts: { include: { contact: { select: { name, platformUserId } } } }` for board rendering
- **System prompt delegation docs**: Full action tag documentation for `add_checklist` with `assigneeType`/`delegateTo`/`dueDate` params, `link_contact` with `involvement` param

## Card Merging
- **merge_cards action tag**: `[[merge_cards:{"targetCardId":"...","sourceCardId":"..."}]]` — merges source INTO target. All tasks, contacts, artifacts move. Source description appended. Source deleted.
- **API endpoint**: `POST /api/kanban/merge` — same logic, used by the UI
- **UI**: 🔀 Merge button in CardDetailModal footer. Shows dropdown of other cards, two-step confirm (warns source will be deleted).
- **Default behavior**: Auto-merge ON by default. Divi tells user when it merges, user can undo via Board. Configurable in Settings → Your Divi → Triage.
- **Auto-routing**: Off by default. Configurable in Settings → Your Divi → Triage. When off, signal items triaged conversationally.

## Due Date Discipline
- **ChecklistItem.dueDate**: DateTime? field added to schema. `add_checklist` action tag accepts `dueDate` ISO string.
- **System prompt rule**: Every task should have a due date. Divi infers from context ("by Friday", "end of month", "ASAP" = today). For tasks with no temporal signal, suggests defaults by priority (urgent=today, high=+2d, medium=+7d, low=+14d) and confirms.

## Two-Tier Fee Model (Payment Routing)
- **Internal transactions** (both parties on same instance): Fee set by `MARKETPLACE_FEE_PERCENT` / `RECRUITING_FEE_PERCENT` env vars. Can be 0% for closed-team deployments.
- **Network transactions** (federation, marketplace, external agents): Enforced minimum floor — 3% marketplace (`NETWORK_MARKETPLACE_FEE_FLOOR`), 7% recruiting (`NETWORK_RECRUITING_FEE_FLOOR`). Payments route through DiviDen.
- Config files: `src/lib/marketplace-config.ts`, `src/lib/recruiting-config.ts` — both export `getInternal*FeePercent()`, `get*FeePercent(isNetworkTransaction)`, `calculate*()`, `get*FeeInfo()` with `networkFeePercent` fields
- Federation validation: `POST /api/v2/federation/validate-payment` — authenticates via platformToken, enforces fee floor, logs telemetry
- All API routes pass `isNetworkTransaction` flag to fee calculators

## Jobs → Paying Tasks Reframe (Apr 13 2026)
- **Core principle**: Jobs are project-based paying tasks, NOT employment. Users are contributors, not workers.
- **Dual projects on acceptance**: Poster gets oversight project, contributor gets execution project (separate boards, linked by job record via `projectId` + `contributorProjectId`)
- **Task breakdown**: `taskBreakdown` field (JSON array of strings) on NetworkJob — becomes kanban cards on contributor's board
- **Link to existing project**: Tasks can reference an existing project via `projectId` — poster's project reused instead of creating new
- **5-star default**: Everyone starts at 5.0⭐. `RATING_THRESHOLD = 5` in job-matcher.ts — real ratings don't factor until 5+ completed tasks
- **Equity/alternative comp**: Supported via freeform `compensation` field. Platform doesn't take a percentage on non-monetary.
- **Language mapping**: Job→Task, Hire→Assign, Worker→Contributor, Client→Poster, Contract→Agreement, Apply→Express Interest, Recruiting fee→Platform fee

## Task Detection & Smart Routing (Apr 13 2026)
- **`propose_task` action tag**: Creates `agent_suggestion` queue item with full task metadata (title, description, taskType, urgency, compensation, skills, estimatedHours, taskBreakdown, sourceCardId, routingSuggestion). Operator reviews in Queue before network posting.
- **Routing priority waterfall**: card contributors → team members → connections → network task board (last resort)
- **Detection triggers in system prompt**: "I need someone to...", scope creep on cards, unmatched delegated items, explicit "find help" requests
- **Decision matrix**: Inner-circle match → `task_route`/`relay_request` (direct). No match → `propose_task` (queue approval). Explicit "post this" → `post_job` (direct). "Find me work" → `find_jobs`.
- **Scoring boosts**: `findSkillMatches` in `brief-assembly.ts` already boosts project members (+10) and team members (+5) — inner-circle-first is baked into the scoring layer
- **Kanban-Driven Orchestration** updated: references inner-circle-first routing and `propose_task`
- **`post_job` updated**: Now supports `projectId` and `taskBreakdown` parameters (was missing from action tag handler)
- **`find_jobs` language**: "matching jobs" → "matching tasks" throughout

## Google OAuth Integration (Apr 12 2026)
- **Purpose**: Data access (signals), separate from NextAuth login. Users authenticate with email/password, then optionally connect Google for Gmail, Calendar, Drive.
- **Google Cloud Project**: `dividen-493203`, Project Number: `689488798650`
- **Scopes**: `gmail.readonly`, `gmail.send`, `gmail.compose`, `calendar` (full read+write), `drive.readonly`, `userinfo.email`, `userinfo.profile`
- **Send route** (`/api/integrations/send`): Detects provider — Gmail API (`users.messages.send` with RFC 2822 raw message) for Google accounts, SMTP/nodemailer for SMTP accounts
- **Redirect fix**: All OAuth routes use `getPublicBaseUrl(req)` helper (reads `x-forwarded-host`) instead of `req.url` to avoid container hostname leaking
- **Gmail scopes are RESTRICTED**: Google security audit required for 100+ users. Testing mode allows up to 100 test users.
- **Redirect URIs** (must be configured in Google Console):
  - `https://dividen.ai/api/auth/callback/google-connect`
  - `https://sdfgasgfdsgsdg.abacusai.app/api/auth/callback/google-connect`
  - `http://localhost:3000/api/auth/callback/google-connect`
- **Files**:
  - `src/lib/google-oauth.ts` — Token exchange, refresh, scope management, `getValidAccessToken()` with auto-refresh (5-min buffer)
  - `src/lib/google-sync.ts` — `syncGmail()`, `syncCalendar()`, `syncDrive()`, `syncAllGoogleServices()` using `googleapis` npm
  - `src/app/api/auth/google-connect/route.ts` — Initiates OAuth, builds consent URL with state param
  - `src/app/api/auth/callback/google-connect/route.ts` — Handles callback, exchanges code, creates 3 IntegrationAccount records (email, calendar, drive)
- **State param**: base64url-encoded JSON `{userId, identity}` for security
- **Callback**: Separate from NextAuth's `/api/auth/callback/google` to avoid conflicts
- **Token storage**: `IntegrationAccount.accessToken`, `refreshToken`, `tokenExpiry`, `scope`, `syncCursor` fields
- **Unique constraint**: `@@unique([userId, identity, service, accountIndex])` — supports multiple Google accounts per identity
- **Multi-inbox**: `accountIndex Int @default(0)` on IntegrationAccount. Operators can connect up to 3 Google accounts (accountIndex 0-2), agents get 1. UI groups accounts under a single Google section with per-account disconnect.
- **Drive sync**: Stores files as Document records with `fileSource: 'google_drive'`, deduplicates via `tags: 'gdrive:<driveId>'`
- **Disconnect**: DELETE with `?disconnectGoogle=true` revokes at Google and deletes all Google integrations for that identity
- **Action tag**: `[[sync_signal:{"service":"email|calendar|drive|all"}]]` — triggers sync from system prompt
- **System prompt**: Integration awareness in Group 4 (business ops layer), sync_signal in action tags reference, sync guidance in Layer 16
- **UI**: IntegrationManager redesigned — "Connect Google" OAuth button per identity, shows connected services with Sync/Disconnect. SMTP form kept as "+ SMTP" secondary option.
- **Self-hosted OAuth isolation**: When `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` missing, `/api/auth/google-connect` returns 501. IntegrationManager shows friendly self-setup prompt with Google Cloud Console link instead of broken connect button. `googleOAuthAvailable` boolean returned from GET `/api/integrations`.
