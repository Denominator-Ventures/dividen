## Onboarding Flow (Project-Based — v2)
- **Philosophy**: Onboarding IS a project. No separate phase-tracking system. Users learn the platform by using it.
- **Welcome popup**: `src/components/dashboard/OnboardingWelcome.tsx` — two-step modal:
  - Step 1: Welcome intro with 3 feature highlights → "Get Started →" button
  - Step 2: BYOAI (Bring Your Own AI) key entry — provider selector (Anthropic/OpenAI), API key input with validation.
  - Accepts `initialStep` prop — dashboard shows step 2 directly if user has `hasSeenWalkthrough=true` but no active API key.
- **Post-API-key flow**:
  1. `POST /api/onboarding/intro` — sends Divi's intro message + "together or solo?" choice widget into chat
  2. User picks "Walk me through it" (together) or "I'll handle it myself" (solo)
  3. `POST /api/onboarding/setup-project` — creates "DiviDen Setup" project with kanban task cards
     - **Together mode**: Tasks due today, Divi walks through each step
     - **Solo mode**: Tasks due in 1 week, framed as "confirm done or need help?"
  4. Setup tasks appear in Now Panel and on the Board as real kanban cards
- **Setup tasks**: Configure Working Style, Set Triage Preferences, Connect Email & Calendar, Review Connected Signals, Custom Signals (optional), Run First Catch-Up
- **No more old 6-phase system**: `onboardingPhase` set to 6 immediately. Old `onboarding/advance` still works for legacy but new users get project-based flow.
- **API key gate**: Dashboard checks `d.data?.apiKeys?.some(k => k.isActive)` — if no active key, shows OnboardingWelcome at step 2 regardless of onboarding state.

## Now Panel vs Queue — Separation of Concerns
- **Now Panel** = operator's priorities (things assigned to ME)
  - Sources: kanban cards (assignee='human') with due dates, goals with deadlines, calendar events, relay responses
  - Queue items are **excluded** — they belong to Divi
  - New task from Now Panel creates a kanban card (assignee='human', status='active')
  - Mark complete on kanban items patches via `/api/kanban/[id]` (status='done')
- **Queue Panel** = Divi's inbox (things for Divi/agents to execute)
  - Sources: all queue items
  - Includes pending_confirmation items that need user approval
- **Never overlap**: The same item should never appear in both panels
- **Tab badges**: CenterPanel shows badge counts for Email (unread), Calendar (events today), Drive (file count) — refreshed every 60s
- **Chat alignment**: User messages RIGHT (brand-colored bubble), Divi messages LEFT (surface-colored bubble with avatar)

## Queue Confirmation Gate & CoS Execution Engine
- **Confirmation gate**: All queue items enter as `pending_confirmation` by default. User must approve before Divi acts.
- **queueAutoApprove**: User-level boolean in User model (default false). When true, items skip confirmation and go straight to `ready`.
- **Action tags**: `dispatch_queue` and `queue_capability_action` in `src/lib/action-tags.ts` check `queueAutoApprove` to set initial status.
- **Confirm endpoint**: `POST /api/queue/confirm` (session-based) and `POST /api/v2/queue/{id}/confirm` (Bearer token) — accepts `{ queueId, action: 'approve'|'reject' }`.
- **Chat-based queue control**: 3 new action tags — `confirm_queue_item` (approve pending→ready), `remove_queue_item` (delete), `edit_queue_item` (update + trigger smart prompter). System prompt instructs Divi to offer confirm/edit/remove after proposing items.
- **Smart Task Prompter v2**: `src/lib/smart-task-prompter.ts` — Agent-aware optimization engine. On edit, it:
  1. Resolves the target agent's Integration Kit from MarketplaceAgent (taskTypes, requiredInputSchema, contextInstructions, usageExamples, executionNotes).
  2. Calls LLM to produce `displaySummary` (≤120 chars for queue card UI) and `optimizedPayload` (full structured payload matching the agent's input schema).
  3. Stores both in `metadata` JSON (never overwrites title/description). Also stores `_original`, `_optimizedAt`, `_optimizedForAgent`.
  4. Falls back to generic `{ task, context, deliverables, files, constraints }` structure when no agent schema is available.
- **Inline edit UI**: QueueItemCard has ✏️ button → inline form (title, desc, priority) → "Save & Optimize" triggers PATCH + smart prompter. Shows "✨ optimizing..." pulse while running. ⚡ badge shown when optimizedPayload exists.
- **Queue card display**: Shows `displaySummary` as headline (if available), full title below if truncated, description up to 2 lines. Full title on hover.
- **CoS relay dispatch**: `cos-sequential-dispatch.ts` sends `optimizedPayload` in relay when available, falls back to raw description.
- **QueuePanel UI**: `src/components/dashboard/QueuePanel.tsx` renders a `pending_confirmation` section with ✅/❌ buttons.
- **CoS mode**: `src/lib/cos-sequential-dispatch.ts` — proactive execution engine. `executeTask()` parses handler to determine strategy (capability/relay/generic), performs the action, logs via `logActivity()`, advances to next task.
- **logActivity()** takes an object: `logActivity({ userId, action, actor, summary })`.
- **v2 Settings API**: `GET/PATCH /api/v2/settings` — reads/writes `mode` and `queueAutoApprove`.
- **OpenAPI docs**: Updated with `pending_confirmation` status, `/queue/{id}/confirm`, `/settings` endpoints.
- **Developer docs**: 3 new sections — Queue Gating & Confirmation, CoS Execution Engine, Settings API v2.
- **Release update**: Entry added to `src/lib/updates.ts` covering all changes.

## Teams Architecture
- **Schema**: `Team` model has `originInstanceUrl` (null = dividen.ai platform) and `isSelfHosted` (bool, default false). Self-hosted teams bypass all billing/subscription gates.
- **TeamInvite model**: Token-based invites with role, message, expiry. Supports email, userId, or connectionId targets.
- **Feature gates**: `src/lib/feature-gates.ts` — all subscription/billing checks bypass for self-hosted teams via `isTeamSelfHosted()`. Returns synthetic unlimited subscription.
- **Pricing boundary**: Platform teams require subscription. Self-hosted teams are free. Billing follows team origin, not member origin.
- **Team→Project sync**: `src/lib/team-project-sync.ts` — syncs team members to project members on assignment and on member join.
- **CoS project delegation**: `cos-sequential-dispatch.ts` `findProjectContributor()` — delegates generic project tasks to qualifying contributors via relay. Works for individual and team-sourced members. Strategy: capability → explicit relay → project contributor → generic.
- **UI**: TeamsView has invite panel, project assignment, self-hosted badge. Invite acceptance at `/team/invite/[token]`.
