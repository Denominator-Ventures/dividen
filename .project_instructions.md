## System Prompt — Modular Capability Architecture (Phase 1)
- **File**: `src/lib/system-prompt.ts`
- **Philosophy**: "Capabilities as installed modules" — every Divi ships with core, optional modules load on-demand by relevance scoring.
- **Old**: Monolithic `buildCapabilitiesAndSyntax()` (7,219 tokens) loaded on EVERY message.
- **New**: 5 modular functions, only `capabilities_core` always loads:
  - `buildCapabilitiesCore()` (~3,200 tokens, ALWAYS) — Card CRUD, Checklist, People, Artifacts, Queue, Goals, Profile/Memory/Setup tags, Interactive Widgets, Linked Kards, Continuous Task Awareness, Core Operating Principles
  - `buildTriageCapabilities()` (~1,200 tokens, ON-DEMAND) — Full 8-step triage protocol, outbound capabilities. Signals: triage, catch-up, morning, signals, email, transcript, comms.
  - `buildRoutingCapabilities()` (~800 tokens, ON-DEMAND) — Task detection, routing waterfall, task board, orchestration, invite intake. Signals: route, assign, delegate, who should, relay, connection.
  - `buildFederationCapabilities()` (~200 tokens, ON-DEMAND) — Entity resolve, serendipity, route_task, network_briefing. Signals: federation, cross-instance, remote, network.
  - `buildMarketplaceCapabilities()` (~200 tokens, ON-DEMAND) — list/execute/subscribe/install/uninstall agent. Signals: marketplace, agent, install, subscribe.
- **Relevance engine**: `PromptGroup` type has 17 groups (was 13). `SIGNAL_PATTERNS` regex map drives `scoreGroupRelevance()`. Groups with empty patterns (like `capabilities_core`) always score 1.0.
- **Setup layer**: `buildSetupLayer_conditional` no longer takes `onboardingPhase`. "Everything configured" path is ~200 tokens (status line + nav ref). "Incomplete" path has widget mappings only.
- **Dead code removed**: `layer16_platformSetupAssistant_optimized` (never called), duplicate Federation Intelligence section.
- **Token savings**: ~5,000-6,000 tokens saved per non-triage/non-routing message.
## CapabilityModule Phase 2 (SHIPPED — v1.6.1)
- **File**: `src/lib/capability-module.ts`
- **Interface**: `CapabilityModule` — formal contract for marketplace capabilities integrating with the relevance engine.
- **Schema additions** to `MarketplaceCapability`: `signalPatterns` (JSON array of regex strings), `tokenEstimate` (int), `alwaysLoad` (bool), `moduleVersion` (string, default "1.0"), `totalGrossRevenue`, `totalPlatformFees`, `totalDeveloperPayout`.
- **Scoring**: `scoreCapabilityModule()` uses same weights as static groups — message match (+0.6), context match (+0.3), baseline (+0.05), threshold (0.3). `alwaysLoad=true` → 1.0.
- **Loading**: `loadRelevantCapabilityModules(userId, message, context)` fetches installed caps, scores them, returns only those above threshold.
- **Prompt injection**: `buildCapabilityModulePrompt()` formats scored modules with metadata for system prompt.
- **System prompt integration**: Injected as Group 14 in `buildSystemPrompt()`. Each module loaded independently — no longer dumped under generic `active_caps`.
- **Revenue**: 97/3 split (same as agents) applied to paid capability purchases via `calculateRevenueSplit()`.
- **API**: `GET /api/v2/prompt-groups` — exposes all 17 static groups, signal patterns, scoring params, CapabilityModule spec. For federation alignment.
- **Webhook**: `POST /api/marketplace/webhook` — receives `agent_approval` events from managed marketplace. Validates `X-Federation-Token`.
