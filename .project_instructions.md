## Onboarding Flow (Project-Based — v2)
- **Philosophy**: Onboarding IS a project. No separate phase-tracking system. Users learn the platform by using it.
- **Welcome popup**: `src/components/dashboard/OnboardingWelcome.tsx` — two-step modal:
  - Step 1: Welcome intro with 3 feature highlights → "Get Started →" button
  - Step 2: BYOAI (Bring Your Own AI) key entry — provider selector (Anthropic/OpenAI), API key input with validation.
  - Accepts `initialStep` prop — dashboard shows step 2 directly if user has `hasSeenWalkthrough=true` but no active API key.
- **Post-API-key flow**:
  1. `POST /api/onboarding/intro` — creates project + 1 kanban card + 6 checklist items + chat messages in a **single $transaction** for speed. All reads parallelized with `Promise.all`.
  2. User picks "Walk me through it" (together) or "I'll handle it myself" (solo)
  3. `POST /api/onboarding/setup-project` — updates due dates on existing checklist items (today vs 1 week), returns firstTask for auto-discussion
  4. Setup tasks appear in Now Panel as checklist_task items and on the Board as 1 card with 6 checklist items
- **Setup structure**: 1 project → 1 kanban card ("DiviDen Setup") in active column → 6 checklist items (each with assigneeType='self', dueDate, order)
- **Setup tasks**: Configure Working Style, Set Triage Preferences, Connect Email & Calendar, Review Connected Signals, Custom Signals (optional), Run First Catch-Up
- **No more old 6-phase system**: `onboardingPhase` set to 6 immediately. Old `onboarding/advance` still works for legacy but new users get project-based flow.
- Setup project cards appear naturally in Divi's kanban board context (group 2) with `proj:"DiviDen Setup"` label — no special system prompt block needed.
- Divi uses existing `[[complete_checklist:{"id":"..."}]]` to mark tasks done and `[[update_card:{"id":"<id>","status":"completed"}]]` when all done.
- `getCompletionMessage` removed entirely — no hardcoded "Setup Complete" screen ever exists.
- Legacy onboarding phases (0-5) still work for old-flow users.
- **API key gate**: Dashboard checks `d.data?.apiKeys?.some(k => k.isActive)` — if no active key, shows OnboardingWelcome at step 2 regardless of onboarding state.
- **Auto-install capabilities**: When Google is connected (email/calendar), `src/app/api/auth/callback/google-connect/route.ts` silently upserts AgentCapability records for 'email' (Outbound Email) and 'meetings' (Meeting Scheduling) with sensible default rules. User discovers them later in Capabilities tab.

## Cockpit Mode — Work Partner Behavior
- **Default behavior**: When user opens chat, Divi proactively works through their NOW list (assigned checklist tasks + active cards).
- **System prompt** now includes `myChecklistTasks` — the operator's assigned incomplete checklist items with due dates, fetched in the batch query.
- **Task lifecycle in chat**: Divi picks highest-priority item → helps execute → marks complete via `[[complete_checklist:{"id":"..."}]]` → creates follow-on tasks → moves to next.
- **Activity logging**: `complete_checklist`, `send_email`, `create_calendar_event` action tags now log to `activityLog` with `action: 'capability_executed'` or `'task_completed'`.
- **Capability execution from chat**: When context is clear and low-risk, Divi can execute capabilities (email, meetings) directly from chat without going through the queue. Logged as activity.
- **CoS mode** (Chief of Staff): Divi works autonomously through the task list, coordinating with other agents via comms and using installed capabilities. Queue still used for high-stakes items.
- **Three assignment types**: "self" (operator does it), "divi" (agent handles via queue/capabilities), "delegated" (routed to another user's Divi via relay)

## Board Cortex — Intelligent Board Analysis Layer
- **File**: `src/lib/board-cortex.ts` — Pure functions for board intelligence
- **API**: `GET/POST /api/board/cortex` — GET returns digest, POST triggers full scan with auto-housekeeping
- **Prisma model**: `BoardInsight` — stores detected issues (merge_suggestion, stale_card, auto_escalate, duplicate_tasks, archive_candidate) with confidence scores
- **Functions**:
  - `detectDuplicates(cards)` — Levenshtein similarity (75% threshold) across card titles
  - `detectDuplicateTasks(cards)` — Cross-card checklist item similarity (80% threshold)
  - `findStaleCards(cards, now)` — Active cards with no update for 14+ days
  - `findEscalationCandidates(cards, now)` — Cards within 48h of deadline with <30% completion → auto-bumped to urgent
  - `findArchiveCandidates(cards, now)` — Completed cards 2+ days old
  - `computeBoardHealth(cards, now)` — Summary stats
  - `buildContextDigest(userId, cards, now)` — Pre-digested summary for system prompt
  - `runBoardScan(userId)` — Full scan: detect + persist insights + auto-escalate + log activity
- **System prompt integration**: `buildContextDigest` called during prompt construction, output injected as "🧠 Board Intelligence" section in Group 2 (Active State). Supplements raw card dump with TOP FOCUS, BOARD HEALTH, RECENT COMPLETIONS, and BOARD INTELLIGENCE (actionable merge/escalate/stale suggestions).
- **Reuses**: `similarity()` from `queue-dedup.ts` (Levenshtein)
- **Philosophy**: Board should be self-cleaning; Divi only sees the sharpest version of reality. LLM's job is judgment and execution, not data janitoring.

## Linked Kards v2 — Cross-User Card Visibility with Auto-Linking & Status Propagation
- **Schema**: `CardLink` model — bidirectional link between two KanbanCards (fromCardId → toCardId)
- **CardLink enriched fields**: `linkedStatus`, `linkedPriority` (cached from linked card), `lastSyncedAt`, `externalCardId` + `externalInstanceUrl` (FVP cross-instance)
- **KanbanCard provenance fields**: `originCardId` (sender's card), `originUserId` (delegator), `sourceRelayId` (delivery relay)
- **AgentRelay**: `cardId` field — direct FK to source card (replaces JSON-buried card context)
- **File**: `src/lib/card-links.ts` — `linkCards()`, `getLinkedCards()`, `getLinkedCardsForUser()`, `formatLinkedCardsForPrompt()`, `autoLinkFromRelay()`, `propagateCardStatusChange()`
- **Auto-linking (v2)**: `autoLinkFromRelay()` called after every card creation — finds recent inbound relay with `assign_task` intent and `cardId`, stamps provenance + creates link automatically. No LLM action required.
- **Manual override**: `linkedFromCardId` param still works on `create_card`/`upsert_card` as priority 1 override
- **Explicit linking**: `link_cards` action tag — `[[link_cards:{"fromCardId":"...","toCardId":"...","linkType":"collaboration"}]]`
- **Status propagation (v2)**: `propagateCardStatusChange()` fires on all status change points (update_card, archive_card, kanban PATCH, card-auto-complete). Updates CardLink cached status + sends update relay to originator.
- **Relay→Card wiring**: `task_route` stamps `cardId: context.card.id` on relay creation. `relay_request` accepts `cardId` param.
- **System prompt**: Group 2 board context shows ⬅️delegated-from:username for provenance + 🔗 prefix for linked cards
- **Kanban API**: `GET /api/kanban` returns `linkedCards` array + `originCardId`/`originUserId`/`originUserName`/`sourceRelayId` on each card
- **UI**: KanbanView shows purple delegation provenance badge + linked card indicators with direction, type, title, user, and checklist progress

## Now Panel vs Queue — Separation of Concerns
- **Now Panel** = operator's priorities (things assigned to ME)
  - Sources: kanban cards (assignee='human') with due dates, goals with deadlines, calendar events, relay responses
  - Queue items are **excluded** — they belong to Divi
  - New task from Now Panel creates a kanban card (assignee='human', status='active')
  - Mark complete on kanban items patches via `/api/kanban/[id]` (status='done')
- **Queue Panel** = Divi's inbox (things for Divi/agents to execute)
  - Sources: all queue items
  - Includes pending_confirmation items that need user approval
- **Never overlap**: The same item should never appear in both panels
- **Tab badges**: CenterPanel shows badge counts for Email (unread), Calendar (events today), Drive (file count) — refreshed every 60s
- **Chat alignment**: User messages RIGHT (brand-colored bubble), Divi messages LEFT (surface-colored bubble with avatar)

## Queue Confirmation Gate & CoS Execution Engine
- **Confirmation gate**: All queue items enter as `pending_confirmation` by default. User must approve before Divi acts.
- **queueAutoApprove**: User-level boolean in User model (default false). When true, items skip confirmation and go straight to `ready`.
- **Action tags**: `dispatch_queue` and `queue_capability_action` in `src/lib/action-tags.ts` check `queueAutoApprove` to set initial status.
- **Confirm endpoint**: `POST /api/queue/confirm` (session-based) and `POST /api/v2/queue/{id}/confirm` (Bearer token) — accepts `{ queueId, action: 'approve'|'reject' }`.
- **Chat-based queue control**: 3 new action tags — `confirm_queue_item` (approve pending→ready), `remove_queue_item` (delete), `edit_queue_item` (update + trigger smart prompter). System prompt instructs Divi to offer confirm/edit/remove after proposing items.
- **Smart Task Prompter v2**: `src/lib/smart-task-prompter.ts` — Agent-aware optimization engine. On edit, it:
  1. Resolves the target agent's Integration Kit from MarketplaceAgent (taskTypes, requiredInputSchema, contextInstructions, usageExamples, executionNotes).
  2. Calls LLM to produce `displaySummary` (≤120 chars for queue card UI) and `optimizedPayload` (full structured payload matching the agent's input schema).
  3. Stores both in `metadata` JSON (never overwrites title/description). Also stores `_original`, `_optimizedAt`, `_optimizedForAgent`.
  4. Falls back to generic `{ task, context, deliverables, files, constraints }` structure when no agent schema is available.
- **Inline edit UI**: QueueItemCard has ✏️ button → inline form (title, desc, priority) → "Save & Optimize" triggers PATCH + smart prompter. Shows "✨ optimizing..." pulse while running. ⚡ badge shown when optimizedPayload exists.
- **Queue card display**: Shows `displaySummary` as headline (if available), full title below if truncated, description up to 2 lines. Full title on hover.
- **CoS relay dispatch**: `cos-sequential-dispatch.ts` sends `optimizedPayload` in relay when available, falls back to raw description.
- **QueuePanel UI**: `src/components/dashboard/QueuePanel.tsx` renders a `pending_confirmation` section with ✅/❌ buttons.
- **CoS mode**: `src/lib/cos-sequential-dispatch.ts` — proactive execution engine. `executeTask()` parses handler to determine strategy (capability/relay/generic), performs the action, logs via `logActivity()`, advances to next task.
- **logActivity()** takes an object: `logActivity({ userId, action, actor, summary })`.
- **v2 Settings API**: `GET/PATCH /api/v2/settings` — reads/writes `mode` and `queueAutoApprove`.
- **OpenAPI docs**: Updated with `pending_confirmation` status, `/queue/{id}/confirm`, `/settings` endpoints.
- **Developer docs**: 3 new sections — Queue Gating & Confirmation, CoS Execution Engine, Settings API v2.
- **Release update**: Entry added to `src/lib/updates.ts` covering all changes.

## Teams Architecture
- **Schema**: `Team` model has `originInstanceUrl` (null = dividen.ai platform) and `isSelfHosted` (bool, default false). Self-hosted teams bypass all billing/subscription gates.
- **TeamInvite model**: Token-based invites with role, message, expiry. Supports email, userId, or connectionId targets.
- **Feature gates**: `src/lib/feature-gates.ts` — all subscription/billing checks bypass for self-hosted teams via `isTeamSelfHosted()`. Returns synthetic unlimited subscription.
- **Pricing boundary**: Platform teams require subscription. Self-hosted teams are free. Billing follows team origin, not member origin.
- **Team→Project sync**: `src/lib/team-project-sync.ts` — syncs team members to project members on assignment and on member join.
- **CoS project delegation**: `cos-sequential-dispatch.ts` `findProjectContributor()` — delegates generic project tasks to qualifying contributors via relay. Works for individual and team-sourced members. Strategy: capability → explicit relay → project contributor → generic.
- **UI**: TeamsView has invite panel, project assignment, self-hosted badge. Invite acceptance at `/team/invite/[token]`.
