# DiviDen Command Center — Project Instructions

## Overview
DiviDen is an individual-first, opencore personal AI command center. Dark theme, command-center aesthetic. Deployed at dividen.ai and sdfgasgfdsgsdg.abacusai.app.

## Key Design Decisions
- **Publisher Attribution**: All platform items published under "DiviDen" (`publisherType: 'platform'`). User-submitted capabilities get `publisherType: 'user'` with their name.
- **Agent Skills (agentskills.io)**: Capabilities can be flagged as Agent Skills via `skillFormat`, `skillBody`, `skillSource` fields. Displayed with 🧩 badge.
- **Approval Flow**: User-submitted capabilities start as `pending_review`. Admin can approve/reject with reason. Only `approved` items visible in marketplace browse.
- **Inbox → Email**: Renamed throughout (CenterPanel, KeyboardNav, GlobalSearch).
- **Smart Task Assembly**: QueuePanel uses multi-step task creation (type → objective → context → review) instead of freeform input.
- **Network Auto-Connect**: Users on dividen.ai primary instance are auto-connected. Federation wizard hidden for primary instance.
- **Settings Loading**: All settings pages (Profile, Relay, Divi) have retry logic (2 retries, 1s delay) to handle transient DB connection pool exhaustion.

## Admin Panel
- Admin auth uses Bearer token with ADMIN_PASSWORD env var for all admin APIs
- Admin capabilities API: `/api/admin/capabilities` (CRUD with Bearer auth)
- Admin marketplace API: `/api/admin/marketplace` (expanded with POST/DELETE for agents)
- MarketplaceTab has 4 sub-tabs: Agents | Capabilities | + Capability | + Agent
- Users fetched from `/api/admin/stats` → `users.list` for publish-as-user dropdowns

## AmbientSurveys Agent
- Marketplace agent seeded via `scripts/seed.ts` as `MarketplaceAgent` with slug `ambient-surveys`
- Published by DiviDen, $5/person surveyed (per_task pricing model)
- Free access password: `freeme`

## Dynamic System Prompt
- `buildSystemPrompt` now accepts `currentMessage` in context
- Context Relevance Engine scores 13 prompt groups against current message + last 3 messages
- Always loads: identity, state, conversation, capabilities
- Other groups (people, memory, schedule, relay, extensions, setup, business, team, active_caps) loaded only when message signals relevance
- Short/greeting messages load all groups as fallback
- Adds a `Dynamic context` note to the prompt when not all groups are loaded

## Phased Onboarding System
- **Phase tracking**: `onboardingPhase` Int on User model (0=api-key, 1=divi-settings, 2=email-setup, 3=connected-overview, 4=webhooks, 5=propagation, 6=complete)
- **Phase 0**: Existing OnboardingWizard (API key + provider setup), shown when `onboardingPhase === 0`
- **Phases 1-5**: Chat-based, driven by interactive widgets rendered inline in Divi's messages
- **Phase definitions**: `src/lib/onboarding-phases.ts` — messages, widget configs per phase
- **Advance API**: `POST /api/onboarding/advance` — generates phase-specific Divi message with widget metadata, saves to chat, advances phase
- **Phase API**: `GET/POST /api/onboarding/phase` — check/set current phase
- **Interactive widgets**: `OnboardingChatWidgets.tsx` — sliders, toggles, radio groups, selects, Google OAuth connect buttons, webhook setup, submit/skip buttons
- **ChatView integration**: `MessageBubble` detects `isOnboarding` metadata and renders `OnboardingChatWidgets` inline
- **Hidden messages**: User messages starting with `[SYSTEM:` are hidden from display
- **Re-entry**: On login, if `onboardingPhase` is 1-5, dashboard auto-resumes by generating the current phase message
- **Google OAuth during onboarding**: `returnTo=onboarding` param flows through OAuth state, redirects back to `/dashboard?google=connected`
- **Skip**: Every phase has a skip button, which advances to next phase without applying settings
- **Settings API**: Returns `onboardingPhase` in user data for dashboard phase detection

## Gemini Meeting Notes Integration
- `src/lib/gemini-meeting-notes.ts` — generates structured meeting notes using Gemini 2.0 Flash
- Works with transcript (from recordings) or without (context-only from event metadata)
- API endpoint: `POST /api/recordings/meeting-notes` (requires eventId, optional recordingId)
- Action tag: `[[generate_meeting_notes:{"eventId":"...","recordingId":"optional"}]]`
- Saves notes as Document (type: meeting_notes), updates Recording summary, links to card artifacts
- Requires `GEMINI_API_KEY` env var (placeholder set, needs actual key from user's Google Cloud project)
- System prompt auto-detects when Google Calendar is connected + Gemini configured and advertises capability
