## Smart Triage System (Task-First Architecture)
- **Mental Model**: Cards = Projects (containers), Checklist Items = Tasks (atomic work items). Every signal item produces tasks, not cards.
- **Triage Protocol** (system-prompt.ts): 8-step Task-First Routing — EXTRACT TASKS → ROUTE TO PROJECT → ADD TO EXISTING (checklist + artifact) → CREATE NEW PROJECT (sparingly, name as initiative not task) → ASSIGN → QUEUE → LEARN → SUMMARIZE
- **upsert_card** action tag: Levenshtein similarity matching (≥80%) — finds existing card by title before creating
- **link_artifact** action tag: Dual-write system — sets direct FK for built-in types (email, document, recording, calendar_event, contact, comms) AND creates generic CardArtifact record. Custom signal types use CardArtifact only (extensible to ANY artifact type string).
- **CardArtifact model**: Generic join table (cardId, artifactType, artifactId, label, metadata) with unique constraint. Supports any custom signal artifact type (slack_message, github_pr, notion_page, etc.)
- **ChecklistItem source tracking**: sourceType, sourceId, sourceLabel fields — traces WHERE each task came from (which signal, which artifact)
- **Board rendering**: Shows per-card task delegation breakdown `[me:2 divi:3 via-divi:1]`, contributor names with 🟢 for DiviDen users, related count, artifact counts (📧📄🎙️📅👤💬🔗)
- **Signal definitions**: Both `cardTypes` (legacy) and `taskTypes` (new) arrays. UI shows taskTypes. All triage prompts follow EXTRACT → ROUTE → CREATE pattern.
- **Custom signals**: Auto-generate task-first triage prompts with signal-specific artifactType for [[link_artifact:{}]]
- **Queue dedup** (queue-dedup.ts): Levenshtein ≥80% prevents duplicate queue items

## Task Delegation & People Model
- **Assignee types on ChecklistItem**: `assigneeType` field — `self` (operator marks complete), `divi` (AI handles directly), `delegated` (routed to another user's Divi agent)
- **Delegation metadata**: `assigneeName`, `assigneeId` (platformUserId of delegate), `delegationStatus` (pending/accepted/in_progress/completed/declined)
- **Display pattern**: Delegated tasks show as "Name via Divi" so the board feels human, not just "me and AI"
- **People on cards**: `CardContact.involvement` — `contributor` (can take tasks, shown by name) vs `related` (contextual only, shown as count)
- **DiviDen user detection**: `CardContact.canDelegate` auto-set from `contact.platformUserId` presence. 🟢 = DiviDen user who can receive delegated tasks
- **Delegation flow**: Operator → Divi (project manager) → target user's Divi agent (via relay). Divi orchestrates, never bypasses.
- **Kanban query includes contacts**: `contacts: { include: { contact: { select: { name, platformUserId } } } }` for board rendering
- **System prompt delegation docs**: Full action tag documentation for `add_checklist` with `assigneeType`/`delegateTo`/`dueDate` params, `link_contact` with `involvement` param

## Card Merging
- **merge_cards action tag**: `[[merge_cards:{"targetCardId":"...","sourceCardId":"..."}]]` — merges source INTO target. All tasks, contacts, artifacts move. Source description appended. Source deleted.
- **API endpoint**: `POST /api/kanban/merge` — same logic, used by the UI
- **UI**: 🔀 Merge button in CardDetailModal footer. Shows dropdown of other cards, two-step confirm (warns source will be deleted).
- **System prompt rule**: NEVER auto-merge. If Divi detects overlap, suggest merge and wait for operator confirmation.
- **No auto-routing to board**: Signal items are triaged conversationally — Divi never silently adds to the board.

## Due Date Discipline
- **ChecklistItem.dueDate**: DateTime? field added to schema. `add_checklist` action tag accepts `dueDate` ISO string.
- **System prompt rule**: Every task should have a due date. Divi infers from context ("by Friday", "end of month", "ASAP" = today). For tasks with no temporal signal, suggests defaults by priority (urgent=today, high=+2d, medium=+7d, low=+14d) and confirms.
