## Federation Push & Ack Loop (v2.1.3+)
- `src/lib/federation-push.ts` — three exported functions:
  - `pushRelayToFederatedInstance()` — POSTs to remote `/api/federation/relay`, on 200 ack: logs activity, stores `peerRelayId`, posts comms confirmation
  - `pushRelayAckToFederatedInstance()` — POSTs completion/decline back to originating instance's `/api/federation/relay-ack`
  - `pushNotificationToFederatedInstance()` — lightweight notification push (project invites, etc.)
- `/api/federation/relay-ack` (NEW) — inbound completion callback from remote instance. On receipt:
  1. Updates relay status → completed/declined
  2. Logs activity (`federation_relay_completed`)
  3. Posts comms message to sender
  4. Advances queue item → `done_today`, triggers sequential dispatch
  5. Updates checklist item delegation status
  6. Checks card auto-completion
  7. Fires webhook `relay_state_changed`
- `relay_respond` action tag: now pushes ack back to originating instance if relay has `peerRelayId`
- `PATCH /api/relays/[id]`: same ack-back wiring for UI-driven completions
- Sending Divi owns the loop — every push expects an ack, every completion flows back

## Comms Peer Name Resolution
When showing thread peer names, NEVER use positional `conn.requester`/`conn.accepter` directly — the logged-in user could be either side. Always compare `.id` against `userId` and pick the OTHER side. Pattern used in both `CommsTab.tsx` and `comms/page.tsx`.

## Notification Center Architecture
- Feed API (`/api/notifications/feed`) aggregates raw activity logs into summaries
- Noise actions filtered: `board_scan`, `sync_completed`, `setup_action`, `queue_status_changed`
- High-signal actions always shown individually: `connection_accepted`, `task_dispatched`, `learning_generated`, etc.
- Low-signal actions grouped by `action + hour` window → "3 tasks queued for dispatch"
- User prefs stored in `User.notificationPrefs` (JSON): `{ enabled: bool, categories: { board: bool, ... } }`
- Preferences API: `GET/PUT /api/notifications/preferences`
- Shared helpers in `src/lib/notification-prefs.ts`
- Settings > Alerts tab has two sections: Notification Center prefs + Cockpit Alert rules

  - Multi-user state observation across sessions
- **Test workflow per bug**:
  1. Jon pastes a single prompt to Divi

  3. Divi lists any new bugs/edge cases it notices during the test
  4. If receive-side verification needed, Divi tells Jon what to look for after switching to Jaron/FVP
  5. Jon confirms or files a follow-up
- **Every debug pass going forward** should produce:
  1. A test prompt per bug (Divi-executable)
  2. Expected pass criteria
  3. Manual verification steps (only for UI-only bugs)
