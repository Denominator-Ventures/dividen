## Onboarding
- NOW tasks order: Chat with Divi (#1) → Connect email (#2) → Analyze inbox (#3) → Setup email capability (#4) → Approve first outbound email (#5) → Connect calendar (#6) → Setup meetings capability (#7) → Add contact (#8) → Set goal (#9) → Teach rule (#10) → Marketplace (#11) → Invite collaborator (#12)
- Flow: email triage → capability setup → outbound queue actions → THEN encourage outside agents

## Capabilities System
- `AgentCapability` model: type (email/meetings/custom), identity (operator/agent/both), rules (JSON[]), config (JSON), status (setup/enabled/disabled/paused)
- API: `/api/capabilities` (GET list, POST upsert), `/api/capabilities/[type]` (GET, PATCH, DELETE)
- CapabilitiesView: ⚡ tab in CenterPanel with setup wizard (4 steps: intro → identity → rules → review)
- Action tag: `[[queue_capability_action:{}]]` creates queue items with `capabilityType` metadata
- QueuePanel: Detects capability metadata, shows ✉️/📅 icons, Approve/Review/Skip buttons
- System prompt Group 13: Dynamic capabilities context injected from DB

## Signals & Catch Up
- Signal framework: `src/lib/signals.ts` — 6 built-in signals (email, calendar, recordings, crm, drive, connections)
- `SignalConfig` model: per-user priority (int), catchUpEnabled (bool), triageEnabled (bool) per signal
- `CustomSignal` model: user-created webhook/integration signals with signalId, triagePrompt, webhookSecret
- API: `/api/signals/config` (GET merged configs, PUT bulk update), `/api/signals/custom` (GET list, POST create)
- `getCatchUpPrompt(configs?)` — dynamic prompt respecting priority order and exclusions
- `handleCatchUp` in dashboard fetches signal config before building prompt
- CatchUpSettings component: modal with drag-to-reorder priority, per-signal catch-up/triage toggles
- Accessible from: ⚙ button next to Catch Up in header, and "Catch Up Settings" link in Signals tab

## Federation & Network APIs (new)
- `GET /api/v2/updates` — Public unified updates feed (no auth, CORS enabled, cacheable)
- `GET /api/v2/network/discover` — Public network discovery (profiles, teams, agents). Optional Bearer token for richer data.
- `POST /api/v2/federation/register` — Self-hosted instance registration → returns platformToken
- `POST /api/v2/federation/marketplace-link` — Enable/disable marketplace participation (requires platformToken)
- `POST /api/v2/federation/heartbeat` — Periodic health check from federated instances
- InstanceRegistry model extended: platformLinked, platformToken, marketplaceEnabled, discoveryEnabled, updatesEnabled, version, userCount, agentCount, lastSyncAt
- FederationManager UI: "Connect to Network" wizard with pre-flight checks, feature toggles, and token display
